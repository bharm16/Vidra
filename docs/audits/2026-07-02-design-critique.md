# Design Critique: Vidra — Full-Product Audit (2026-07-02)

**Method.** Live walkthrough of the Vite client at 1440×900 (plus 390×844 spot-checks) with the API mocked at the browser edge via Playwright route interception — the GCP suspension blocks real server boot. Mock payloads follow the real wire contracts (canonical `ApiResponse` envelopes; NDJSON span stream with required `confidence`; `previewEnvelope` for frames). 25 UI states captured to `.playwright-mcp/audit/audit-*.png`. Four parallel review passes (visual hierarchy, usability flow, visual consistency, design-token code audit), then every load-bearing finding root-caused in source. Accessibility was **excluded by instruction** — no contrast/WCAG/touch-target/aria findings appear here.

**Mock caveats.** The frame image is an SVG stand-in (frame _chrome_ judged, not content). The expansion mock always returns the same lighthouse text, so the prompt/title mismatch in `audit-18` is an artifact. The "Score: 25%" value was computed against mock data — the toast itself fires unconditionally in product code. Sessions-panel contents came from localStorage, not the mocked API.

---

## Overall Impression

The core canvas is genuinely strong: the dark stage makes the frame the hero, the composer is pixel-stable across every state, and the review-gate copy ("Does this frame match your idea?") is exactly the right register for non-experts. Two systemic problems undermine it. First, **the golden path's second half is broken as designed**: the namesake click-to-enhance interaction is unreachable by construction, and the frame→motion handoff instructs users to use a field that doesn't exist. Second, **the product speaks with two visual voices** — two parallel token vocabularies in code, five primary-button treatments, four error grammars, and light-theme surfaces that appear at exactly the moments trust is most fragile (errors).

## Visual Hierarchy

- **What draws the eye first:** on the canvas screens, correctly the frame — the dark stage defers to the artwork (best thing about the design). On first-run, the 44px "What are you making?" lands well.
- **Where hierarchy fails:**
  - **Two lit primaries at the gate** (`audit-07`): "Looks right" and "Make it" are both white fills — step 2 and step 4 presented as equals, making the quality gate skippable by eye. Ghost "Make it" until the gate is answered.
  - **The phase change is a caption swap** (`audit-07`→`09`): after approval, ~95% of pixels are identical; emphasis never moves to the new task. Animate weight downward: dim the frame a step, light the motion tray and CTA.
  - **The gate question whispers** (~15px) while first-run got 44px — typographic guidance collapses at the first real decision. ~20px semibold for phase questions.
  - **Pending frame is a void**: a tiny spinner in a ~718×400 dark box, with the charming "Painting your first frame…" caption exiled below it. Move the caption inside the box; add a shimmer so it reads "working."
  - **Failure emphasis is inverted** (`audit-18`): "No frame yet" (neutral, useless) is centered and biggest; "Couldn't create a frame" + "Try again" (the event and the action) are small text below, and "Try again" is the quietest button in the app on the screen that most needs it.
  - **Marketing hero whispers**: ~26px H1 vs the app's 44px, zero imagery anywhere on the marketing site (the only screen in the product with no visual evidence of the product), ~60% dead viewport below 190px of text, and three co-equal CTAs where the only brand-accent button is the smallest.
- **Reading order:** composer anchored bottom throughout is a reliable spine; template chips vs settings pills share one costume (starter prompts and settings are different behaviors wearing the same pill).

## What Works Well

- **The empty-canvas-to-frame floor is genuinely low**: type one line, hit one button, sensible defaults (16:9 / 10s / Sora 2), ⌘↵ for those who want it.
- **Review-gate copy is exactly right**: "Does this frame match your idea?" / "Looks right" / "Try a different frame" — jargon-free, decision-shaped, placed before money is spent.
- **The expansion moment lands**: one gray sentence becoming a rich multi-color paragraph communicates "Vidra did craftwork" before a word is read.
- **Dark canvas as theater**: the frame is always the brightest object; the environment defers to the artwork.
- **Pixel-stable composer** across all eight captured states — the workspace has a reliable anchor.
- **Model-picker language** describes models by benefit ("Cinematic motion and high fidelity"), and the Render/Draft split matches a novice's mental model.
- **"Painting your first frame…"** is warm, plain-language pending narration.
- **The Asset Library empty state** explains purpose (@triggers) — proof the team can write this; it just needs to spread to the panels.

## Priority Recommendations

1. **Repair the frame→motion handoff as one unit** (U2 + U3 + hierarchy): motion-mode composer (cleared, focused, motion placeholder), stage-specific button labels, chips that insert text, and "Make it" ghosted until the gate is answered. Every user must survive this step; today the instruction points at a field that doesn't exist and the ambiguity is expensive.
2. **Make click-to-enhance reachable — or stop advertising it** (U1 + U8): decide the I2V-mode story, wire the (already-built) suggestion stack to a state users can actually reach, then add the affordance + legend so the colors mean something. It's the namesake interaction and the only recovery path when a frame is close-but-wrong.
3. **One accent, one lit action per state** (hierarchy + visual C-table + C2): decide the accent deliberately — the token system already voted neutral/off-white; the lavender is marketing-only; the purple Generate gradient is stale residue — then give exactly one control per state the primary treatment ("Make it" first-run, "Looks right" at the gate, motion CTA after, "Try again" on failure). This single rule resolves the dual-primary gate, the invisible handoff, the buried retry, and the five-way button fragmentation.
4. **One failure grammar with a next step** (U4 + U7 + U11 + C3): plain-language error copy with one suggested action, primary-weight retry, validate-at-boundary so raw Zod/TypeError output never renders, retheme the boundaries dark, and fix the /history crash with a working remedy.
5. **Merge the token dialects** (C1 + C2 + C5): `--tool-*` → aliases of `--ps-*`, sweep the six stale hexes, and delete PopoverDetail's parallel span palette. Cheap, mechanical, and it makes every future theme decision single-sourced.
